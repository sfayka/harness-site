Open source control plane

Harness verifies completion instead of trusting AI claims.

Why it exists

Execution is not completion.

Agents can report success. Commands can exit cleanly. A branch can contain code. None of that is enough on its own when the real question is whether the task is complete, trustworthy, and ready to count.

Harness is a control plane for that distinction. It treats completion as a governed decision backed by evidence, current task state, reconciliation facts, and explicit review paths.

What Harness does

Canonical evaluation

Harness evaluates task submissions and completion claims against current state instead of accepting agent output as truth.

Evidence-backed completion

Completion depends on artifacts, coherent run context, and the lifecycle rules that govern the task.

Governed reconciliation

Recoverable defects can be repaired and sent back through reevaluation instead of being mistaken for progress.

Explicit review boundaries

When automation should stop, Harness escalates instead of pretending the work is done.

How it works

01

Ingress

Task envelopes arrive from manual, Linear, or executor-adjacent inputs.

02

Completion claim

Claims are treated as inputs to evaluate, not as completion truth.

03

Evaluation and reconciliation

Harness checks evidence, current-run context, and mismatch or recovery conditions.

04

Decision

The task completes, fails, reconciles, or escalates to review.

Design rule: spend automation before operator attention.

Proof

KNO-174

Recovery blocked

Missing PR after execution leads to reconciliation, recovery is blocked, and the task escalates into review instead of being accepted as complete.

Outcome: in review, not silent success.

KNO-175

Recovery succeeds

Harness repairs the missing artifact, attaches evidence, reevaluates, and only then allows the task to complete.

Outcome: completed through reevaluation.

Current scope

What exists now

  • Python backend for canonical evaluation
  • Read-only Next.js dashboard
  • File and Postgres persistence
  • Completion-claim interception
  • Governed reconciliation for missing PR after execution

What this site does not claim

  • That every reconciliation class is automated
  • That the dashboard is a full mutation surface
  • That Harness is a generic chatbot or PM tool
  • That all integrations are already mature sync systems
